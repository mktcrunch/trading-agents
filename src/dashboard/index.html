Twin Ledger is our live experiment, not a backtest, not a replay of history. Two competing traders start with the same capital and compete head-to-head on separate Alpaca accounts, each backed by its own multi-agent crew (coordinator, data, signal, and risk specialists). Gemini still makes the final trade call, but with private forecast context Baseline never sees. This scoreboard is the experiment: who adapts faster as the market moves?

Two Traders, Specialist Crews, Full Traceability

Twin Ledger is a live head-to-head trading experiment on Google Cloud. Two competing traders, Baseline and Internal, each start with $100,000 on a dedicated Alpaca paper account. Every trader is a Vertex AI Agent Engine deployment: a chat-mode coordinator routes scheduler work to task-mode specialists and deterministic execution tools (phrase-matched callbacks, not LLM orchestration). Gemini 3.5 Flash runs on the Vertex global endpoint for signal decisions, discovery planning, chat Q&A, and the intraday trailing-stop planner inside the deterministic risk path. Each overnight run refreshes per-agent learning memory for the full crew before new signals. Every action is logged to GCS with a trace_id and surfaced in this dashboard; equity charts come from Alpaca portfolio history.

Internal consumes MarketCrunch’s production ensemble: trained at scale over 50M+ parameters and 1B+ datapoints. Twin Ledger does not retrain models; it pulls nightly forecasts and injects them into Internal’s data, signal, sizing, and risk agents. Baseline is the control arm and never calls this API.
  • What flows in: per-ETF direction, expected move, and confidence from MarketCrunch /analyze
  • How agents use it: enriched Gemini signal prompts, confidence-based sizing, and an intraday prediction gate on exits
  • What stays the same: identical ADK crew, scheduler, execution path, and audit trail as Baseline
Scheduler → deterministic path. Cloud Scheduler sends a plain-text message (“Run daily trading workflow.”, “Run intraday risk check.”, or “Run post-open chase.”) to each Agent Engine. A before_agent_callback intercepts these phrases and invokes the matching tool directly, with no multi-turn LLM orchestration. If the coordinator is used interactively and does not complete a daily job, an after_agent_callback fallback runs the same deterministic pipeline (DAILY_COORDINATOR_FALLBACK=true).
  • Reliability: unattended cron must complete every time; LLM orchestration can stop after data fetch or explanation without placing orders.
  • Latency & cost: risk runs every 15 minutes; routing should not add extra Gemini turns.
  • Auditability: fixed pipeline (data → signal → risk → execute → monitor) with one trace per job; judges can replay the same steps.
  • Where Gemini stays: signal decisions, discovery planning, dashboard chat (read-only), intraday trailing-stop planner (inside deterministic risk path); only scheduler routing is non-LLM.
  • Learning refresh: before each overnight signal pass, audit outcomes are scored per crew role and persisted to GCS (learning/{role}_{system}.json).

Risk & exit mechanics (intraday)

Baseline risk profile

  • Path: deterministic scheduler → run_intraday_risk_check
  • LLM inside path: Gemini trailing-stop planner each cycle
  • Base stop: fixed −1% return from entry
  • Trailing: pure LLM planner (activation 0.5-4%, lock 45-80%)
  • Prediction gate: none; stops execute immediately
  • EOD exit: losers below −0.95% in final ~6 min before close

Internal risk profile

  • Path: same deterministic scheduler path as Baseline
  • LLM inside path: Gemini trailing planner + hybrid scripted floor (1%/70%)
  • Base stop: ATR × 1.5 dynamic (fallback −1%)
  • Prediction gate: MC Internal short-horizon prediction must agree before stop sell
  • EOD exit: same −0.95% / ~6 min window as Baseline
  • Held exits: logged as risk_held when prediction disagrees

Post-open chase rules

  • Scan overnight DAY/OPG limits placed since prior close
  • Skip if >70% filled or already chased
  • 30-min volatility gate (std dev + range) must be calm
  • Cancel remainder → market order; audit skip reason if gated
What each competing trader does differently

Baseline · System A

Technicals + pure LLM reasoning · 6-agent specialist crew
  • Agents: Coordinator routes data → signal → risk → execution → monitor (no Discovery)
  • Signal: RSI, MACD, Bollinger, news, Google Search grounding, public data only
  • Sizing: LLM size_pct → equal-weight allocation (max 10%)
  • Overnight: DAY limit orders, chased if unfilled
  • Trailing stops: Pure LLM-planned activation & profit lock
  • Base stop: Fixed −1% return threshold
  • Prediction gate: None; exits execute immediately
  • Edge hypothesis: Structured LLM reasoning beats gut feel

Internal · System B

Same public stack as Baseline + MC Internal IP
  • Agents: Same 6-role crew + separate Discovery agent (MC Internal IP only)
  • Signal: Same public context as Baseline, plus MarketCrunch forecasts and vendor features
  • Sizing: confidence-based allocation from the prediction snapshot (max 10% per position)
  • Overnight: Same execution path; prediction snapshot passed to sizing layer
  • Trailing stops: Scripted 1%/70% floor + LLM refinement (hybrid)
  • Base stop: ATR × 1.5 dynamic stop (fallback −1%)
  • Prediction gate: MC Internal defers exit when short-horizon prediction disagrees
  • Edge hypothesis: MC Internal layer beats Baseline when public context is held equal
Agent learning loop
Audit → reflection → next run. After each overnight pipeline start (and on manual refresh), every crew role gets a 7-day scorecard from the audit log. Lessons persist to GCS and appear in the Decisions tab learning panel. Signal and Risk agents receive injected prompt blocks on the next Gemini call.
  • Tier 1, Analyze: role-specific scorecards (e.g. signal: logged / pending / scored decisions; execution: placed vs skipped)
  • Tier 2, Reflect: rule-based heuristics for all roles; optional Gemini polish for Signal + Risk only
  • Tier 3, Inject: build_signal_learning_block and build_risk_learning_block appended to live prompts
  • Pending vs scored: same-day decisions show decisions_pending until next-day bars exist; zero scored ≠ inactive pipeline
